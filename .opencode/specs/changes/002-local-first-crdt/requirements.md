# Требования: Local-first CRDT для Minesweeper

## 1. Общее описание

Реализация local-first подхода для синхронизации состояния игры Minesweeper между несколькими клиентами с использованием CRDT (Conflict-free Replicated Data Types) на базе библиотеки Yjs.

### Ключевые принципы
- Поле игры не хранится в Yjs целиком (слишком большой объём для полей 10000×10000)
- В Yjs хранятся только операции (действия игроков)
- Каждый клиент реконструирует состояние поля, применяя операции к сгенерированному полю
- Детерминированная генерация поля через seed обеспечивает идентичность начального состояния
- **CRDT режим работает только с `mode: 'seeded'`**

---

## 2. Типы операций

### 2.1 JoinGame

Операция присоединения к игре. Первая операция `join` в последовательности создаёт новую игру, последующие — присоединяют новых игроков.

**Параметры:**
- `type`: `'join'`
- `roomId`: string — идентификатор комнаты
- `playerId`: string — ID игрока
- `width`: number — ширина поля
- `height`: number — высота поля
- `minesNum`: number — количество мин
- `seed`: string — seed для генерации поля
- `timestamp`: number — монотонный timestamp

**Особенности:**
- **Первая операция `join`** создаёт новую игру с указанными параметрами
- **Последующие операции `join`** от других игроков означают присоединение к существующей игре
- Параметры игры (`width`, `height`, `minesNum`, `seed`) берутся из **первой** операции `join`
- При получении операции `join` от другого игрока игра **перезапускается** с параметрами из первой `join`
- Затем применяются все остальные операции (`leftClick`, `rightClick`)

### 2.2 LeftClick
Открытие ячейки левой кнопкой мыши.

**Параметры:**
- `type`: `'leftClick'`
- `x`: number — координата X
- `y`: number — координата Y
- `playerId`: string — ID игрока
- `timestamp`: number — монотонный timestamp

### 2.3 RightClick
Постановка/снятие флага правой кнопкой мыши.

**Параметры:**
- `type`: `'rightClick'`
- `x`: number — координата X
- `y`: number — координата Y
- `playerId`: string — ID игрока
- `timestamp`: number — монотонный timestamp

### 2.4 Важное замечание о рекурсивном открытии

При клике на ячейку без соседних мин (пустая ячейка с "0"), в классическом сапёре автоматически открываются соседние ячейки — это называется **рекурсивное открытие** или **flood fill**.

**В рамках CRDT-подхода:**
- В Yjs документ записывается **только одна операция** — клик по конкретной ячейке (x, y)
- Рекурсивное открытие соседних ячеек **НЕ сохраняется** как отдельные операции
- Каждый клиент выполняет рекурсивное открытие **локально и независимо** на основе:
  - Сгенерированного поля (одинакового для всех благодаря seed)
  - Полученной операции клика
  - Собственной логики flood fill

**Пример:**
```
Игрок кликает на пустую ячейку (5, 5) — в Yjs сохраняется:
{ type: 'leftClick', x: 5, y: 5, playerId: 'p1', timestamp: 123 }

Клиент 1 локально открывает: (5,5), (4,4), (4,5), (4,6), (5,4), (5,6), (6,4), (6,5), (6,6)...
Клиент 2 получает операцию и тоже локально выполняет тот же flood fill
```

**Преимущества такого подхода:**
- Минимальный размер операций в Yjs
- Нет необходимости синхронизировать десятки/сотни открытых ячеек
- Каждый клиент вычисляет конечное состояние самостоятельно
- Детерминированность благодаря одинаковому seed поля

## 3. Структура Yjs документа

### Вариант A: Y.Array

```typescript
// Структура: Y.Array<Operation>
const operations = doc.getArray('operations')
operations.push([{
  type: 'leftClick',
  x: 10,
  y: 20,
  playerId: 'player-1',
  timestamp: 1234567890
}])
```

**Достоинства:**
- Простая семантика — операции добавляются в конец
- Естественный порядок операций (хронологический)
- Эффективное добавление новых операций (O(1))
- Простая итерация для реконструкции состояния

**Недостатки:**
- Дедупликация требует полного прохода по массиву
- Поиск дубликатов: O(n) для каждой новой операции
- Удаление операций (если потребуется) — O(n)

### Вариант B: Y.Map с ключом-тimestamp

```typescript
// Структура: Y.Map<Operation> где ключ = `${playerId}-${timestamp}-${x}-${y}`
const operations = doc.getMap('operations')
const key = `${playerId}-${timestamp}-${x}-${y}`
operations.set(key, {
  type: 'leftClick',
  x: 10,
  y: 20,
  playerId: 'player-1',
  timestamp: 1234567890
})
```

**Достоинства:**
- Мгновенная дедупликация (Y.Map перезаписывает по ключу)
- Проверка на дубликат: O(1)
- Удобный доступ к конкретной операции по ключу
- Автоматическая обработка одновременных изменений

**Недостатки:**
- Не гарантирует порядок при итерации (нужна сортировка)
- Ключ может быть длинным (зависит от данных)
- Менее интуитивная структура для лога операций

### Выбранное решение
Использовать **Вариант A (Y.Array)** как основную структуру хранения операций.

**Обоснование:**
- Естественный хронологический порядок операций
- Простота реализации и отладки
- Эффективное добавление новых операций
- Дедупликацию можно реализовать через Set с уже обработанными ключами операций

> **Примечание:** Вариант B (Y.Map) оставлен для истории — может быть рассмотрен при оптимизации производительности с большим количеством операций.

## 6. Правила разрешения конфликтов (Merge Rules)

### 6.1 Дедупликация
Две операции считаются дубликатами, если:
- Одинаковый `type` (leftClick/rightClick)
- Одинаковые координаты (`x`, `y`)
- `playerId` может быть разным

**Результат:** Дубликат пропускается, применяется только первая операция.

### 6.2 Приоритет операций и обработка дубликатов

**Если клетка уже открыта локально:**
- Приходящая операция (внешняя) **полностью пропускается**
- Ничего не выполняется, игнорируется

**При конфликте на одной ячейке:**
- **LeftClick (открытие)** имеет приоритет над **RightClick (флаг)**
- Если один игрок поставил флаг, а другой открыл ячейку — побеждает открытие
- После открытия флаг не может быть установлен

### 6.3 Порядок применения
Порядок операций не критичен для игровой логики:
- Открытие соседних пустых ячеек коммутативно
- Открытие мины завершает игру независимо от порядка
- Применять операции отсортированными по timestamp для детерминизма

## 7. Процесс присоединения к игре (Join Game)

### 7.1 Алгоритм присоединения

1. **Игрок вызывает `joinGame(roomId)`**
   - Передает `roomId` комнаты для присоединения
   - Устанавливается `gameStatus = 'Pending'` в ожидании синхронизации

2. **Подключение к Yjs документу (async)**
   - Провайдер синхронизации подключается к существующему документу
   - Ожидаем получения всех операций через синхронизацию (`await`)

3. **Поиск первой операции `join`**
   - Находим первую операцию `type: 'join'` в массиве операций
   - Извлекаем параметры: `width`, `height`, `minesNum`, `seed`, `roomId`
   - **Важно:** Параметры игры берутся ИЗ первой операции `join`, а не передаются при вызове

4. **Перезапуск игры (`restartGame`)**
   - Очищаем текущее состояние (`reset`)
   - Yjs документ очищается
   - Устанавливаем новые параметры из первой `join`
   - Генерируем поле с тем же `seed` (детерминированно)

5. **Добавление операции `join`**
   - Создаем и добавляем в Yjs операцию `type: 'join'` с текущими параметрами
   - Сигнализируем о присоединении нового игрока

6. **Применение предыдущих операций**
   - Проходим по всем операциям в массиве (кроме первой `join`)
   - Применяем `leftClick`/`rightClick` для восстановления состояния
   - Порядок важен: каждая операция изменяет состояние поля

7. **Игра готова**
   - Устанавливается `gameStatus = 'Playing'`
   - Новый игрок видит текущее состояние поля
   - Может делать ходы (операции синхронизируются)

### 7.2 Особенности реализации

- **Параметры из первой `join`:** `width`, `height`, `minesNum`, `seed`, `roomId` берутся из первой операции
- **Детерминизм:** Одинаковый `seed` = одинаковое поле на всех клиентах
- **Последовательность:** Сначала перезапуск (с очисткой Yjs), затем добавление своей `join`, затем применение всех операций
- **Асинхронность:** Метод `joinGame` возвращает `Promise<void>`, использует `await` для синхронизации
- **gameStatus:** Переводится в `'Pending'` при начале присоединения и в `'Playing'` после восстановления состояния

## 8. Сценарий Game Over

1. Игрок кликает на мину (LeftClick)
2. GameEngine локально обрабатывает поражение
3. Операция добавляется в Yjs документ
4. Другие клиенты получают операцию через синхронизацию
5. Каждый клиент применяет операцию — игра завершается

**Примечание:** Рестарт игры в текущей задаче не рассматривается.

## 9. Варианты сохранения (SaveManager)

### Вариант А: Чистый CRDT подход
Сохраняем только мета-информацию и лог операций.

```typescript
interface SaveDataA {
  version: 2
  meta: {
    width: number
    height: number
    minesNum: number
    seed: string
    roomId: string
    createdAt: number
  }
  operations: Operation[] // Из Yjs документа
}
```

**Преимущества:**
- Минимальный размер сохранения
- Полная совместимость с CRDT подходом
- Восстановление состояния через replay операций
- Естественная поддержка многопользовательского режима

**Недостатки:**
- Медленная загрузка при большом количестве операций
- Необходимость полного replay при загрузке
- Возможные проблемы производительности на 10,000+ операциях

### Вариант Б: Гибридный подход
Сохраняем мета, лог операций и текущий ArrayBuffer.

```typescript
interface SaveDataB {
  version: 2
  meta: {
    width: number
    height: number
    minesNum: number
    seed: string
    roomId: string
    createdAt: number
  }
  operations: Operation[]
  boardState?: ArrayBuffer // Опционально — только для малых полей
}
```

**Преимущества:**
- Быстрая загрузка для малых полей (есть готовое состояние)
- Fallback на replay для больших полей
- Гибкость в зависимости от размера поля

**Недостатки:**
- Больший размер сохранения для малых полей
- Сложность логики (два пути восстановления)
- Возможная рассинхронизация boardState и operations

### Выбранное решение
Использовать **Вариант А** (чистый CRDT подход) — сохраняем только мета-информацию и массив операций.

**Обоснование:**
- Минимальный размер сохранения (критично для передачи по сети)
- Не нужно передавать состояние 100 миллионов клеток
- Полная совместимость с CRDT подходом
- Простота реализации и отладки

> **Примечание:** Вариант Б (гибридный подход) оставлен для истории — может быть рассмотрен при оптимизации загрузки для малых полей.

### 9.1 Процесс загрузки игры

Алгоритм загрузки идентичен процессу присоединения к игре:

1. **Чтение файла** — десериализация бинарного формата .mine
2. **Применение Yjs state update** — восстановление операций в документе
3. **Поиск первой операции `join`** — извлечение параметров игры
4. **Перезапуск игры** — `restartGame()` с параметрами из первой `join`
5. **Применение операций** — последовательный `replay` всех leftClick/rightClick

**Важно:**
- Параметры игры (width, height, minesNum, seed) всегда берутся из первой операции `join`, а не из JSON metadata
- JSON metadata используется только для проверки и совместимости
- Операции применяются в строгом порядке для восстановления состояния

### 9.2 Обратная совместимость

- **Миграция с версии 1 не поддерживается**
- При попытке загрузить файл версии 1 выводится ошибка о несовместимости формата
- Старые сохранения не восстанавливаются, пользователь видит сообщение об ошибке

## 11. Архитектура тестирования

### Тестовый сценарий: Два инстанса

```typescript
// Псевдокод теста
const roomId = generateRoomId()

// Инстанс 1 (Host)
const player1 = new GamePlayer('Alice')
const engine1 = new GameEngine({
  mode: 'seeded',
  roomId,
  playerId: player1.id
})

// Инстанс 2 (Client)
const player2 = new GamePlayer('Bob')
const engine2 = new GameEngine({
  mode: 'seeded',
  roomId,
  playerId: player2.id
})

// EventBus для синхронизации
const eventBus = new EventBus()
eventBus.on('sync', (op) => {
  // Передача операций между инстансами
})
```

### Тестовые кейсы

1. **Базовая синхронизация**
   - Игрок 1 открывает ячейку
   - Операция синхронизируется с игроком 2
   - Состояние идентично на обоих клиентах

2. **Конфликт флаг vs открытие**
   - Игрок 1 ставит флаг
   - Игрок 2 открывает ту же ячейку
   - Результат: ячейка открыта (приоритет открытия)

3. **Дедупликация**
   - Одинаковая операция от того же игрока дважды
   - Применяется только один раз

4. **Offline mode**
   - Игрок 1 делает 5 кликов
   - Синхронизация приостановлена
   - Синхронизация возобновлена
   - Игрок 2 получает все 5 операций и применяет их

5. **Game Over синхронизация**
   - Игрок 1 взрывается
   - Игрок 2 получает операцию взрыва
   - Игра завершена на обоих клиентах

## 12. Ограничения и допущения

### В рамках текущей задачи
- Нет UI для отображения действий других игроков
- Нет ограничения на количество участников
- Нет идентификации игроков (будет отдельная задача)
- Нет рестарта игры (кроме перезапуска при join)
- Нет снапшотов состояния
- Производительность при 10,000+ операциях — не критична

### Будущие улучшения (не в текущей задаче)
- WebSocket / WebRTC транспорт
- Идентификация и авторизация игроков
- Визуализация действий других игроков
- Система рестарта и новых раундов
- Оптимизация для большого количества операций

## 14. Критерии приёмки

- [ ] Два инстанса GameEngine с одним roomId синхронизируют операции
- [ ] Конфликт "флаг vs открытие" разрешается в пользу открытия
- [ ] Дубликаты операций не применяются повторно
- [ ] Game Over синхронизируется между всеми клиентами
- [ ] **Присоединение к игре работает корректно** — второй игрок получает параметры из первой join операции и видит текущее состояние поля
- [ ] **Загрузка игры работает корректно** — при загрузке из файла восстанавливается состояние через перезапуск с параметрами из первой join операции и replay всех операций
- [ ] Сохранение и загрузка работают с новым форматом версии 2
- [ ] Файлы версии 1 не загружаются (выводится ошибка)
- [ ] Все существующие тесты проходят
- [ ] Новые интеграционные тесты на CRDT проходят
