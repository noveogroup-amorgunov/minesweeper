# Требования: Local-first CRDT для Minesweeper

## 1. Общее описание

Реализация local-first подхода для синхронизации состояния игры Minesweeper между несколькими клиентами с использованием CRDT (Conflict-free Replicated Data Types) на базе библиотеки Yjs.

### Ключевые принципы
- Поле игры не хранится в Yjs целиком (слишком большой объём для полей 10000×10000)
- В Yjs хранятся только операции (действия игроков)
- Каждый клиент реконструирует состояние поля, применяя операции к сгенерированному полю
- Детерминированная генерация поля через seed обеспечивает идентичность начального состояния
- **CRDT режим работает только с `mode: 'seeded'`**
- **GameRoom является единой точкой входа** для всех игровых операций (single-player и multiplayer)

---

## 2. Архитектура: GameRoom как центральная абстракция

### 2.1 Разделение ответственности

**GameRoom отвечает за:**
- Создание и присоединение к игровой комнате
- Управление игроком (текущий playerId, name)
- Синхронизацию через CrdtManager
- Маршрутизацию операций к GameEngine
- Подписку на события синхронизации от провайдера
- Публичный API для UI (единая точка входа)

**GameEngine отвечает за:**
- Игровую логику (поле, открытие ячеек, флаги)
- Генерацию поля через Web Worker
- Определение победы/поражения
- Минимальные изменения для поддержки внешних операций

**GameEngine НЕ знает о:**
- CRDT и Yjs
- Сетевой синхронизации
- Других игроках
- isLocal / isRemote различиях

## 3. Типы операций

### 3.1 JoinGame

Операция присоединения к игре. Первая операция `join` в последовательности создаёт новую игру, последующие — присоединяют новых игроков.

**Параметры:**
- `type`: `'join'`
- `roomId`: string — идентификатор комнаты
- `playerId`: string — ID игрока
- `width`: number — ширина поля
- `height`: number — высота поля
- `minesNum`: number — количество мин
- `seed`: string — seed для генерации поля
- `timestamp`: number — монотонный timestamp

**Особенности:**
- **Первая операция `join`** создаёт новую игру с указанными параметрами
- **Последующие операции `join`** от других игроков означают присоединение к существующей игре
- Параметры игры (`width`, `height`, `minesNum`, `seed`) берутся из **первой** операции `join`
- При получении операции `join` от другого игрока игра **перезапускается** с параметрами из первой `join`
- Затем применяются все остальные операции (`leftClick`, `rightClick`)

### 3.2 LeftClick

Открытие ячейки левой кнопкой мыши.

**Параметры:**
- `type`: `'leftClick'`
- `x`: number — координата X
- `y`: number — координата Y
- `playerId`: string — ID игрока
- `timestamp`: number — монотонный timestamp

### 3.3 RightClick

Постановка/снятие флага правой кнопкой мыши.

**Параметры:**
- `type`: `'rightClick'`
- `x`: number — координата X
- `y`: number — координата Y
- `playerId`: string — ID игрока
- `timestamp`: number — монотонный timestamp

### 3.4 Важное замечание о рекурсивном открытии

При клике на ячейку без соседних мин (пустая ячейка с "0"), в классическом сапёре автоматически открываются соседние ячейки — это называется **рекурсивное открытие** или **flood fill**.

**В рамках CRDT-подхода:**
- В Yjs документ записывается **только одна операция** — клик по конкретной ячейке (x, y)
- Рекурсивное открытие соседних ячеек **НЕ сохраняется** как отдельные операции
- Каждый клиент выполняет рекурсивное открытие **локально и независимо** на основе:
  - Сгенерированного поля (одинакового для всех благодаря seed)
  - Полученной операции клика
  - Собственной логики flood fill

**Пример:**
```
Игрок кликает на пустую ячейку (5, 5) — в Yjs сохраняется:
{ type: 'leftClick', x: 5, y: 5, playerId: 'p1', timestamp: 123 }

Клиент 1 локально открывает: (5,5), (4,4), (4,5), (4,6), (5,4), (5,6), (6,4), (6,5), (6,6)...
Клиент 2 получает операцию и тоже локально выполняет тот же flood fill
```

**Преимущества такого подхода:**
- Минимальный размер операций в Yjs
- Нет необходимости синхронизировать десятки/сотни открытых ячеек
- Каждый клиент вычисляет конечное состояние самостоятельно
- Детерминированность благодаря одинаковому seed поля

## 4. Класс GameRoom

### 4.1 Ответственность

GameRoom является **единой точкой входа** для всех игровых операций. UI работает только с GameRoom, не напрямую с GameEngine.

### 4.2 Конфигурация

```typescript
interface GameRoomConfig {
  /** ID комнаты (опционально — для single-player генерируется автоматически) */
  roomId?: string
  /** Имя игрока (опционально) */
  playerName?: string
  /** Начальные параметры игры (для создания новой игры) */
  gameParams?: {
    width: number
    height: number
    minesNum: number
  }
  /** Фабрика для создания провайдера синхронизации */
  syncProviderFactory?: SyncProviderFactory
  /** Scheduler для задач */
  scheduler: AbstractScheduler
}
```

### 4.3 Публичный API

```typescript
class GameRoom {
  /** Создать новую игру */
  async createGame(params: { width, height, minesNum }): Promise<void>
  
  /** Присоединиться к существующей игре */
  async joinGame(roomId: string): Promise<void>
  
  /** Левый клик по ячейке (вызывается из UI) */
  handleLeftClick(x: number, y: number): void
  
  /** Правый клик по ячейке (вызывается из UI) */
  handleRightClick(x: number, y: number): void
  
  /** Получить текущее состояние игры */
  getGameState(): GameState
  
  /** Подписаться на изменения состояния */
  subscribe(callback: (state: GameState) => void): () => void
  
  /** Получить ID текущего игрока */
  getCurrentPlayerId(): string
  
  /** Получить ID комнаты */
  getRoomId(): string
  
  /** Перезапустить игру (только для single-player) */
  restart(): void
}
```

### 4.4 Жизненный цикл

```
1. Создание GameRoom
   └── Создаётся GamePlayer (текущий игрок)
   └── Создаётся GameEngine с дефолтными параметрами
   └── Создаётся CrdtManager (пустой документ)

2. Создание игры (createGame)
   └── Генерируется roomId
   └── Добавляется операция join в CrdtManager
   └── GameEngine.restart() с переданными параметрами
   └── Подключается SyncProvider (если нужна синхронизация)

3. Присоединение к игре (joinGame)
   └── Подключается к существующему документу через SyncProvider
   └── Получаем все операции из CrdtManager
   └── Находим первую join операцию → параметры игры
   └── GameEngine.restart() с параметрами из первой join
   └── Добавляем свою join операцию
   └── Применяем все leftClick/rightClick через GameEngine.applyOperation()
```

## 6. Изменения в GameEngine

### 6.1 Новый метод applyOperation()

```typescript
// Добавить в GameEngine
applyOperation(op: LeftClickOperation | RightClickOperation): void
```

**Назначение:**
- Применяет операцию от другого игрока (или от себя при replay)
- Не сохраняет операцию в CRDT (это делает GameRoom)
- Использует существующую логику reveal/flag

**Реализация:**
```typescript
applyOperation(op: LeftClickOperation | RightClickOperation): void {
  const index = op.y * this._width + op.x
  
  if (op.type === 'leftClick') {
    this.reveal(index) // Существующий метод
  } else if (op.type === 'rightClick') {
    this.flag(index) // Существующий метод
  }
}
```

### 6.2 Удаление isLocal параметра

Из методов `reveal()` и `flag()` удалить параметр `isLocal` (если он есть). GameEngine не должен знать о различии локальных и удалённых операций.

### 6.3 Метод restart()

GameEngine уже имеет метод `restart()`, используется для перезапуска с новыми параметрами при join.

## 8. Правила разрешения конфликтов (Merge Rules)

### 8.1 Дедупликация

Две операции считаются дубликатами, если:
- Одинаковый `type` (leftClick/rightClick)
- Одинаковые координаты (`x`, `y`)
- `playerId` может быть разным

**Результат:** Дубликат пропускается, применяется только первая операция.

**Реализация в GameRoom:**
```typescript
// GameRoom хранит Set обработанных операций
private processedOps: Set<string>

// Ключ для дедупликации: `${type}-${x}-${y}`
private getOpKey(op: GameOperation): string {
  return `${op.type}-${op.x}-${op.y}`
}
```

### 8.2 Приоритет операций и обработка дубликатов

**Если клетка уже открыта локально:**
- Приходящая операция (внешняя) **полностью пропускается**
- Ничего не выполняется, игнорируется

**При конфликте на одной ячейке:**
- **LeftClick (открытие)** имеет приоритет над **RightClick (флаг)**
- Если один игрок поставил флаг, а другой открыл ячейку — побеждает открытие
- После открытия флаг не может быть установлен

### 8.3 Порядок применения

Порядок операций не критичен для игровой логики:
- Открытие соседних пустых ячеек коммутативно
- Открытие мины завершает игру независимо от порядка
- Применять операции отсортированными по timestamp для детерминизма

## 9. Процесс присоединения к игре (Join Game)

### 9.1 Алгоритм присоединения (внутри GameRoom)

```
1. Игрок вызывает gameRoom.joinGame(roomId)
   └── Устанавливается статус 'Pending' в GameEngine

2. Подключение к Yjs документу (async)
   └── LocalSyncProvider подключается к существующему документу
   └── Получаем все операции через синхронизацию

3. Поиск первой операции join
   └── Находим первую операцию type: 'join' в массиве операций
   └── Извлекаем: width, height, minesNum, seed, roomId
   └── Важно: Параметры игры берутся ИЗ первой операции join

4. Перезапуск игры
   └── Вызываем gameEngine.restart({ width, height, minesNum })
   └── GameEngine использует seed для генерации поля
   └── Поле идентично на всех клиентах благодаря seed

5. Добавление операции join
   └── Создаем операцию type: 'join' с текущим playerId
   └── Добавляем в Yjs документ через CrdtManager

6. Применение предыдущих операций
   └── Проходим по всем операциям в массиве (кроме join)
   └── Для каждой leftClick/rightClick:
       └── Проверяем дедупликацию
       └── Вызываем gameEngine.applyOperation(op)
   └── Восстанавливаем состояние поля

7. Игра готова
   └── Устанавливается gameStatus = 'Playing'
   └── Новый игрок видит текущее состояние поля
   └── Может делать ходы (операции синхронизируются)
```

### 9.2 Особенности реализации

- **Параметры из первой `join`:** `width`, `height`, `minesNum`, `seed`, `roomId` берутся из первой операции
- **Детерминизм:** Одинаковый `seed` = одинаковое поле на всех клиентах
- **Последовательность:** Сначала перезапуск GameEngine, затем добавление своей `join`, затем применение всех операций
- **Асинхронность:** Метод `joinGame()` возвращает `Promise<void>`, использует `await` для синхронизации
- **gameStatus:** Переводится в `'PENDING'` при начале присоединения и в `'PLAYING'` после восстановления состояния

## 10. Сценарий Game Over

1. Игрок кликает на мину (через GameRoom.handleLeftClick)
2. GameRoom добавляет операцию в CrdtManager
3. GameEngine локально обрабатывает поражение
4. Операция синхронизируется с другими клиентами через SyncProvider
5. Другие клиенты получают операцию
6. GameRoom других клиентов применяет операцию через GameEngine.applyOperation()
7. Игра завершается на всех клиентах

**Примечание:** Рестарт игры в текущей задаче не рассматривается.

## 11. Варианты сохранения (SaveManager)

### 11.1 Чистый CRDT подход

Сохраняем только мета-информацию и лог операций.

```typescript
interface SaveData {
  version: 2
  meta: {
    width: number
    height: number
    minesNum: number
    seed: string
    roomId: string
    createdAt: number
    playerId: string
    playerName: string
  }
  operations: Operation[] // Из Yjs документа
}
```

**Преимущества:**
- Минимальный размер сохранения
- Полная совместимость с CRDT подходом
- Восстановление состояния через replay операций
- Естественная поддержка многопользовательского режима

### 11.2 Процесс загрузки игры

Алгоритм загрузки идентичен процессу присоединения к игре:

1. **Чтение файла** — десериализация бинарного формата .mine
2. **Применение Yjs state update** — восстановление операций в документе
3. **Поиск первой операции `join`** — извлечение параметров игры
4. **Перезапуск игры** — `gameEngine.restart()` с параметрами из первой `join`
5. **Применение операций** — последовательный `replay` всех leftClick/rightClick через `gameEngine.applyOperation()`

**Важно:**
- Параметры игры (width, height, minesNum, seed) всегда берутся из первой операции `join`, а не из JSON metadata
- JSON metadata используется только для проверки и совместимости
- Операции применяются в строгом порядке для восстановления состояния

### 11.3 Обратная совместимость

- **Миграция с версии 1 не поддерживается**
- При попытке загрузить файл версии 1 выводится ошибка о несовместимости формата
- Старые сохранения не восстанавливаются, пользователь видит сообщение об ошибке

## 13. Архитектура тестирования

### 13.1 Тестовый сценарий: Два инстанса GameRoom

```typescript
// Псевдокод теста
const roomId = generateRoomId()
const eventBus = new EventBus()

// Инстанс 1 (Host)
const room1 = new GameRoom({
  roomId,
  playerName: 'Alice',
  gameParams: { width: 100, height: 100, minesNum: 1000 },
  scheduler: new Scheduler(),
  syncProviderFactory: (roomId, doc) => 
    new LocalSyncProvider(roomId, doc, eventBus)
})

await room1.createGame({ width: 100, height: 100, minesNum: 1000 })

// Инстанс 2 (Client)
const room2 = new GameRoom({
  playerName: 'Bob',
  scheduler: new Scheduler(),
  syncProviderFactory: (roomId, doc) => 
    new LocalSyncProvider(roomId, doc, eventBus)
})

await room2.joinGame(roomId)
```

### 13.2 Тестовые кейсы

1. **Базовая синхронизация**
   - Игрок 1 открывает ячейку через room1.handleLeftClick()
   - Операция синхронизируется с игроком 2
   - Состояние идентично на обоих клиентах (room1.getGameState() === room2.getGameState())

2. **Конфликт флаг vs открытие**
   - Игрок 1 ставит флаг через room1.handleRightClick()
   - Игрок 2 открывает ту же ячейку через room2.handleLeftClick()
   - Результат: ячейка открыта (приоритет открытия)

3. **Дедупликация**
   - Одинаковая операция от того же игрока дважды
   - Применяется только один раз

4. **Offline mode**
   - Игрок 1 делает 5 кликов
   - Синхронизация приостановлена (eventBus.pause())
   - Синхронизация возобновлена (eventBus.resume())
   - Игрок 2 получает все 5 операций и применяет их

5. **Game Over синхронизация**
   - Игрок 1 взрывается
   - Игрок 2 получает операцию взрыва
   - Игра завершена на обоих клиентах

6. **Присоединение к игре**
   - Создаётся игра с 10 открытыми ячейками
   - Второй игрок присоединяется
   - Второй игрок видит все 10 открытых ячеек

## 14. Ограничения и допущения

### 14.1 В рамках текущей задачи
- Нет UI для отображения действий других игроков
- Нет ограничения на количество участников
- GameRoom хранит только текущего игрока (не список всех игроков)
- Нет рестарта игры (кроме перезапуска при join)
- Нет снапшотов состояния
- Производительность при 10,000+ операциях — не критична

### 14.2 Будущие улучшения (не в текущей задаче)
- WebSocket / WebRTC транспорт (через SyncProvider)
- Идентификация и авторизация игроков
- Список всех игроков в комнате
- Визуализация действий других игроков
- Система рестарта и новых раундов
- Оптимизация для большого количества операций

## 16. Критерии приёмки

### 16.1 GameRoom функциональность
- [ ] GameRoom является единой точкой входа для UI

- [ ] GameRoom.createGame() создаёт новую игру с заданными параметрами
- [ ] GameRoom.joinGame() корректно присоединяется к существующей игре

- [ ] Два инстанса GameRoom с одним roomId синхронизируют операции

- [ ] Дубликаты операций не применяются повторно
- [ ] Game Over синхронизируется между всеми клиентами
