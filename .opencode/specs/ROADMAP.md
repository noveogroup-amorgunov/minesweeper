### Local-first подход 1 (PWA)

Можно с этой штуки начать. Здесь достаточно все просто.

Нужно будет упаковать приложение в PWA для использования сервис-воркеров. Здесь важно, чтобы вся статика, кроме index.js, кэшировалась с помощью сервис-воркера. Но при перегенерации приложения у нас кэши менялись в названии файлов, и все перезагружалось.

То есть, фактически, нам не нужно было выключать сервис-воркер или перезапускать его для того, чтобы получить новую версию игры. В этот раз возможно использовать Workbox, который позволит нам создать всю базовую представление для прогрессивов по EBS.

В общем, это достаточно небольшая и понятная задача для того, чтобы игра работала без интернета после первой загрузки.

### Local-first подход 2 (CRDT)

Вот эту штуку нужно будет пробовать либо второй, либо третий.

С дачи достаточно интересная реализация локалхёс подхода с точки зрения резолва конфликтов и формата хранения самой игры. Здесь прям челлендж на счет того, как правильно хранить.

Сейчас игра хранится в виде массива байтов, ну, array буфер. Для резолва конфликтов нужно хранить её в каком-нибудь формате типа yjs, но опять же для CRD team конфликт resolution что-то там.

Собственно, хранить небольшие поля нет проблем, то есть мы просто создаем yjs объект с нашим стейтом приложения, его синкаем между различными инстансами.

То есть, условно, у нас есть два инстанса: в одном откликаем, он синкается с другим. На текущий момент нужно будет это всё проверить в рамках одного приложения. В будущем это будет работать через WebSocket либо WebRTC.

Собственно, тут нужно, челлендж включается в том, чтобы понять, а как правильно хранить большие данные, например, 100 Мбайт array buffer в формате yjs.
